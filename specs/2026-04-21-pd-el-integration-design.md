# PD and EL Integration — Design Spec

**Date:** 2026-04-21
**Repo:** `trisk.analysis`
**Status:** Proposed — awaiting implementation plan
**Scope for this spec:** S1 (integration functions + aggregates) and S2 (workflow vignette). S3 (PD/EL visualization walkthrough) and S4 (sensitivity vignette simplification) are tracked separately.

---

## 1. Motivation

TRISK recomputes probability of default (PD) using a Merton structural model driven by `debt_equity_ratio`, `volatility`, and DCF-implied equity. It **does not consume** the bank's own internal PD estimate. Two consequences:

1. **Analytical:** the level of TRISK's PD is not comparable to the bank's internal PD. Only the *change* under a climate shock is meaningful.
2. **Numerical:** for firms with low volatility, high equity buffer, or short term, the Merton `d2` grows large enough that `pnorm(-d2)` underflows to zero in double precision, producing baseline PDs of 0 even when the firm has real credit risk.

The Shiny desktop app (`trisk.r.docker/app/modules/mod_integration.R`) already solves this with three PD integration methods (absolute, relative, Basel IRB z-score) and two EL methods (absolute, relative), but the logic is trapped inside Shiny observers and cannot be called from R scripts or vignettes.

This spec ports the methodology into `trisk.analysis` as pure functions, with aggregates and visualizations, so the methodology is reusable from any R workflow and documented through a vignette.

---

## 2. Non-goals

- **Not** changing how TRISK computes PD inside `trisk.model` — the Merton calculation stays as-is.
- **Not** fixing the `term > 5` silent-NA bug in `calc_pd_change_overall()` — that is a separate trisk.model issue.
- **Not** building interactive Plotly visuals — all new plots are static ggplot objects so they work in pkgdown vignettes and PDF reports.
- **Not** including client (Khan Bank) data anywhere in the package. Only `trisk.model::testdata` and `trisk.analysis::testdata` are used in examples and vignettes.

## 4. Components — public API

### 4.1 `integrate_pd()`

```r
integrate_pd(
  analysis_data,
  internal_pd = NULL,
  method = c("absolute", "relative", "zscore"),
  zscore_floor = 1e-4,
  zscore_cap   = 1 - 1e-4
)
```

**Returns** a list:

- `$portfolio` — the input `analysis_data` with five columns added: `internal_pd`, `pd_change`, `pd_change_pct`, `trisk_adjusted_pd`, `pd_adjustment`. Row count unchanged.
- `$portfolio_long` — long-format helper built by pivot-longer on `pd_baseline / pd_shock / internal_pd / trisk_adjusted_pd`. Columns: existing keys + `pd_type` (factor, ordered: `internal`, `baseline`, `shock`, `trisk_adjusted`) + `pd_value`. Used directly by `pipeline_crispy_pd_integration_bars()`.
- `$aggregate` — one-row tibble from `aggregate_pd_integration()` (see 4.3).

**Method math** (ported line-for-line from `mod_integration.R:170-234`):

| Method | Formula |
|---|---|
| absolute | `trisk_adjusted_pd = internal_pd + (pd_shock - pd_baseline)` |
| relative | `trisk_adjusted_pd = internal_pd * (1 + pd_change_pct)` where `pd_change_pct = (pd_shock - pd_baseline) / pd_baseline`, falls back to 0 when `pd_baseline = 0` |
| zscore | `trisk_adjusted_pd = pnorm(qnorm(clip(internal_pd)) + qnorm(clip(pd_shock)) - qnorm(clip(pd_baseline)))` with clipping to `[zscore_floor, zscore_cap]` |

Output floored at 0 and capped at 1.

**`internal_pd` resolution priority** (matches Shiny):
1. If `internal_pd` is a numeric vector of length `nrow(analysis_data)` — use directly.
2. If `internal_pd` is a dataframe with `company_id` + `internal_pd` columns — match by `company_id`; unmatched rows fall back to `pd_baseline`.
3. If `NULL` — fall back to `pd_baseline` for every row.

**Known quirks documented in roxygen (Shiny parity):**
- `relative` method with `pd_baseline = 0`: `pd_change_pct` is 0, so `trisk_adjusted_pd = internal_pd` — shock signal is lost. This is the documented Shiny behavior; users who want a non-zero shock signal on zero-baseline rows should use `absolute` or `zscore`.
- `zscore` method silently clips. Clip bounds are exposed as `zscore_floor` / `zscore_cap` arguments.

### 4.2 `integrate_el()`

```r
integrate_el(
  analysis_data,
  internal_el = NULL,
  method = c("absolute", "relative")
)
```

**Returns** a list:
- `$portfolio` — `analysis_data` + columns `internal_el`, `el_change`, `el_change_pct`, `trisk_adjusted_el`, `el_adjustment`.
- `$portfolio_long` — pivot-longer on `expected_loss_baseline / expected_loss_shock / internal_el / trisk_adjusted_el`.
- `$aggregate` — one-row tibble from `aggregate_el_integration()`.

Method math ported from `mod_integration.R:509-557`. No capping (EL can exceed exposure mathematically and trisk.model may emit negatives depending on LGD setup).

### 4.3 `aggregate_pd_integration()`

```r
aggregate_pd_integration(portfolio_df)
```

**Returns** a one-row tibble:

| Column | Formula |
|---|---|
| `total_exposure_usd` | `sum(exposure_value_usd)` |
| `weighted_pd_internal` | `sum(internal_pd * exposure_value_usd) / total_exposure_usd` |
| `weighted_pd_baseline` | `sum(pd_baseline * exposure_value_usd) / total_exposure_usd` |
| `weighted_pd_shock` | `sum(pd_shock * exposure_value_usd) / total_exposure_usd` |
| `weighted_pd_adjusted` | `sum(trisk_adjusted_pd * exposure_value_usd) / total_exposure_usd` |
| `weighted_pd_adjustment` | `weighted_pd_adjusted - weighted_pd_internal` |
| `weighted_pd_adjustment_pct` | `weighted_pd_adjustment / weighted_pd_internal` (NA if internal is 0) |

Follows the `aggregate_facts_*` convention in `prepare_plot_data.R:8-56`: `group_by_at` (trivial single-group when called at portfolio level) + `summarise(..., na.rm = TRUE, .groups = "drop")`.

Optional `group_cols` argument (default NULL) enables sector-level aggregation when passed `c("sector")` or `c("sector", "technology")`.

### 4.4 `aggregate_el_integration()`

```r
aggregate_el_integration(portfolio_df)
```

**Returns** a one-row tibble:

| Column | Formula |
|---|---|
| `total_exposure_usd` | `sum(exposure_value_usd)` |
| `total_el_internal` | `sum(internal_el)` |
| `total_el_baseline` | `sum(expected_loss_baseline)` |
| `total_el_shock` | `sum(expected_loss_shock)` |
| `total_el_adjusted` | `sum(trisk_adjusted_el)` |
| `total_el_adjustment` | `total_el_adjusted - total_el_internal` |
| `el_adjusted_bps` | `abs(total_el_adjusted) / total_exposure_usd * 10000` |

Optional `group_cols` like PD aggregate.

## 6. Error handling

All errors use `stop()` with descriptive messages. No silent failures except the Shiny-parity `relative` method edge case documented above.

| Condition | Behavior |
|---|---|
| `method` not in allowed set | Hard error via `match.arg()` (default) |
| Required columns missing (`pd_baseline`, `pd_shock` for PD; `expected_loss_*` for EL; `exposure_value_usd` for aggregates) | Hard error with column list |
| `internal_pd` vector length ≠ `nrow(analysis_data)` | Hard error |
| `internal_pd` dataframe missing `company_id` column | Hard error |
| `internal_pd` dataframe has unmatched rows | Silent fallback to `pd_baseline` for unmatched; match count returned invisibly |
| `zscore_floor >= zscore_cap` | Hard error |
| All `internal_pd` values NA after resolution | Hard error |

Input validation runs at function entry before any computation, matching the defensive pattern in `check_portfolio()` at `run_trisk_on_portfolio.R:116-127`.

## 7. Testing

`tests/testthat/test-integrate.R` — minimum 20 tests. Follows the existing testthat structure in `trisk.analysis/trisk_test/`.

**Math correctness:**
- PD × 3 methods × (with internal vector / with internal df / NULL) = 9 cases. Assert new columns present and spot-check math against manual computation on a 3-row synthetic input.
- EL × 2 methods × (with internal / NULL) = 4 cases.

**Edge cases:**
- `pd_baseline = 0` + relative method → `trisk_adjusted_pd = internal_pd` (document the quirk in the test).
- `pd_baseline = 1` + zscore method → clipped to `zscore_cap`, stays finite.
- `internal_pd` all NA → hard error.
- PD output clamped to [0, 1] for extreme internal values (e.g., internal = 0.999 + shock high).

**Aggregates:**
- Exposure-weighted PD matches manual `sum(pd * ead) / sum(ead)` on a 3-row fixture.
- Total EL is sum of rows.
- `el_adjusted_bps` correctly scales to bps.
- `group_cols = "sector"` returns correct sector rollup.

**Argument validation:**
- Unknown method string errors.
- Missing column errors.
- Vector length mismatch errors.
- Dataframe missing `company_id` errors.

**Plots (smoke tests):**
- Each `pipeline_crispy_*` returns a `ggplot` object (or `knitr_kable` for P3/P4).
- Calling with zero-row input errors cleanly, not cryptically.

## 8. Workflow vignette — `vignettes/pd-el-integration.Rmd`

Target length: 150–200 lines. Follows the style of `portfolio-analysis.Rmd`: `kableExtra` tables, `knitr::kable() %>% kable_styling(bootstrap_options = c("striped", "hover", "condensed")) %>% scroll_box(...)` for all data display.

**Section outline:**

1. **Setup** — `library(trisk.analysis)`, `library(magrittr)`, load bundled testdata (`assets_testdata.csv`, `scenarios_testdata.csv`, `financial_features_testdata.csv`, `ngfs_carbon_price_testdata.csv` from `trisk.model`; `portfolio_ids_testdata.csv` from `trisk.analysis`).
2. **Run TRISK** — `analysis_data <- run_trisk_on_portfolio(...)`. Show the `pd_baseline / pd_shock / expected_loss_*` columns.
3. **Why integration matters** — one narrative paragraph: TRISK's PD level is not comparable to bank internal PDs; only the *change* is meaningful. Three methods translate TRISK's shift into the bank's own PD.
4. **Method 1 — Absolute** — `result_abs <- integrate_pd(analysis_data, method = "absolute")`; print `$portfolio` head, `$aggregate`.
5. **Method 2 — Relative** — same pattern; call out the zero-baseline quirk in a `note` block.
6. **Method 3 — Z-score (Basel IRB)** — brief Vasicek explanation (2–3 sentences); show result.
7. **Supplying your own internal PDs** — construct a vector (e.g., rep(0.03, nrow(analysis_data))) and re-run with `internal_pd = my_pds, method = "zscore"`.
8. **Visual comparison of methods (N1)** — `pipeline_crispy_pd_method_comparison(analysis_data)`.
9. **Integration charts (P1, P2)** — `pipeline_crispy_pd_integration_bars()`, `pipeline_crispy_el_adjustment_bars()`.
10. **Portfolio-level KPI tables (P3a, P3b)** — `pipeline_crispy_pd_kpi_table(result_abs$aggregate)` and (after the EL section) `pipeline_crispy_el_kpi_table(el_result$aggregate)`.
11. **Sector breakdown (P4)** — `pipeline_crispy_el_sector_breakdown_table()`.
12. **EL integration** — parallel short section with `integrate_el()` + `aggregate_el_integration()`.

No Khan Bank data, no client data. Only bundled package testdata.

## 9. Coding-style conformance

Matching conventions verified in `trisk.analysis/R/{plot_pd_term_plot,prepare_plot_data,imports,run_trisk_on_portfolio}.R` and `trisk.model/R/{calc_pd_change_overall,run_trisk}.R`:

- snake_case function names with action prefix (`integrate_*`, `aggregate_*`, `prepare_for_*`, `draw_*`, `pipeline_crispy_*`).
- Roxygen2 doc blocks with `@param`, `@return`, `@export` for public functions. Internal helpers unexported.
- Native pipe `|>` (R 4.1+, matches existing code).
- `.data$col` pronoun via rlang for tidy evaluation.
- `dplyr::`, `tidyr::`, `ggplot2::`, `stats::`, `scales::` namespaced calls inside functions (no `library()` inside functions).
- Orchestrator → `prepare_*` → `draw_*` decomposition for every new plotting function.
- Color constants from `imports.R`.
- Theme via `TRISK_PLOT_THEME_FUNC()`.
- `scales::percent_format(scale = 100)` for PD y-axes.
- No new package dependencies in `DESCRIPTION`.
